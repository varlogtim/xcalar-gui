Understanding dataflows and batch dataflows

Xcalar Design provides you with built-in functions to cleanse data and transform data to a useful format. In addition, you can use UDFs to extend the Xcalar Design features to suit your data analysis methodology. Running these functions interactively enables you to see results almost instantaneously. However, doing so does not generate results for the entire data source. To gain insight from the entire data source instead of a sample of your data, you must run a batch dataflow.

Understanding dataflows

When you use Xcalar Design to design an algorithm for data analysis, a dataflow is created automatically. It consists of a series of elementary functions executed and tables created when you use Xcalar Design interactively for modeling. You can view the dataflows leading up to each active table so that you can visually trace the origin and movement of data over time. For information about determining data lineage, see the description about showing the schema for a table in Temporary and active tables.

EXAMPLE: You pull three columns from your data source to create a table and use the Smart Type Casting feature to change the data type, and then you perform a Filter operation on one column. The dataflow shows that the table undergoes one Map operation to change the data type and one Filter operation, and each operation results in a temporary table. The active table, named airlines2#h9838, is shown as the last table in the dataflow. The following screenshot illustrates the dataflow graph for airlines2#h9838.

NOTE: The table icons in a dataflow graph are usually blue. A red table icon represents a table containing only erroneous rows. For more information about erroneous rows, see Creating a table with only erroneous rows.

If you want to re-create an active table by changing some of the operations in the dataflow, follow the instructions in Editing a dataflow.

For information about tasks that you can perform on a dataflow, see Performing dataflow tasks and creating batch dataflows.

Understanding batch dataflows

While a dataflow graph enables you to visually trace data lineage and to recreate tables through editing operations, you cannot run the sequence of operations against the entire data source. To do so, you must save the dataflow under a name. A saved dataflow is called a batch dataflow. You can create a batch dataflow for an active table or for a temporary table that exists in a dataflow.

Batch dataflows are shared among all users on the cluster.

Batch dataflows offer these advantages:

  • You can run the set of operations in the dataflow on demand or on a specified schedule. As the data in your data source changes, you can develop insights from your data source over time.
  • The set of operations are run against the entire data source.
  • Because the operations in a dataflow are automated, you can avoid human errors that might happen if you had to manually execute the operations.

  • The batch dataflow is permanent in the sense that the operations can be performed even after you drop a temporary table in the dataflow. Suppose you drop a temporary table to release memory, you can still create a batch dataflow from the dataflow where the table is dropped.

    NOTE:  While you can see the icon for the dropped temporary table in the batch dataflow graph, the table is not put back to the workbook, and you cannot access it in Xcalar Design.
  • If you drop an active table, the dataflow associated with the table is also removed from Xcalar Design. Table dropping cannot be undone, but if you have created a batch dataflow for this table, you can run the dataflow at any time to re-create the table.
  • You can export selected columns from the table or the entire table created by a batch dataflow to another application. Alternatively, you can export the result of the batch dataflow to a table, which automatically appears in your worksheet.

Tables from which you can create a batch dataflow

You can save the dataflow of an active table as a batch dataflow. You can also create a batch dataflow from the steps leading up to a temporary table in a dataflow graph. Understanding the batch dataflow containing multiple dataflows

When you create a batch dataflow, you have the option of saving multiple dataflows under one batch dataflow name. Typically, you want to include multiple dataflows in one batch dataflow for the following reasons:

  • Having one batch dataflow ensures that data from related dataflows is generated and exported to various locations at the same time. You do not have to run individual dataflows one by one.
  • You can customize the operations in each dataflow in the batch dataflow. Values assigned to the parameters are applied uniformly to all the dataflows. Assuming that you intend to use the same parameter values across all dataflows, this approach helps you avoid using inconsistent parameter values. For information about customization, see Customization of dataflow operations.

In the following example, you can create a batch dataflow named airlines3Data to contain three dataflows: one from the active table (airlines3#uW162) and two from the temporary tables (airlines3#uW160 and airlines3#uW161). You can customize the export operation for each dataflow. When you start airlines3Data, all dataflows are run, and the results of the dataflows are saved to your specified locations, which must be a folder, not a table.

For information about tasks associated with batch dataflows, see Using batch dataflows.
