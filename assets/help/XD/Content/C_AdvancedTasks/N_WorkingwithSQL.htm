Working with SQL in Xcalar Design

This section describes how to use SQL in Xcalar Design for manipulating your data for analysis.

Understanding Xcalar SQL

Xcalar SQL enables you to run relational database query type processing on your data in Xcalar Design. The Xcalar SQL is based on the traditional SQL ANSI standard and T-SQL.

The Xcalar Design table publishing abstraction SQL extension creates a schema based alias from your Xcalar table. It publishes the alias (SQL Table Alias) into the SQL namespace providing support for SQL query statements and operations on structured and semi-structured data. You can also combine Xcalar Design’s extensive operations with your SQL query statement operations.

The SQL Table Aliases are listed in the Tables and Schemas available to SQL section of the Xcalar Design SQL panel, which provides an easy way to manage your SQL Table aliases and their schemas.

From the SQL code editor in the Xcalar Design SQL panel, you enter and run a SQL query statement using the SQL Table Alias name as the table name, which compiles your query, runs the relevant operation, and returns your results in a resultant Xcalar table.

The dataflow graph of your resultant Xcalar table enables you to view the operations that were executed, including the SQL query statements that were applied to your data. For more information, see Viewing SQL operations from the dataflow graph.

Relationship between a Xcalar table and a SQL Table Alias

The SQL Table Alias is an alias created for a Xcalar table that enables you to use SQL query statements that are case-insensitive on a Xcalar table which uses case-sensitive naming conventions. The alias is associated with a schema that contains the column names and data types, such as integer or string, of the originating Xcalar table.

TIP: When a SQL Table Alias has been created from a Xcalar table, the new resultant Xcalar table displays the table name and column names in all capital letters.

Creating a SQL Table Alias

You can only create a SQL Table Alias on an active Xcalar table.

To create a SQL Table Alias:

  1. Create a Xcalar table by doing one of the following:
    • From your source data, create a dataset and then created a table from this dataset.
    • From an existing dataset, create a table.

    Alternatively, use an existing active Xcalar table.

  2. From the sidebar, select the Extensions icon .

    3. The My Extensions panel opens.

  3. Locate the SQL extension and then select Create SQL Table Alias.

    4. The SQL:Create SQL Table Alias section opens.

  4. From the Table list, select the Xcalar table that contains the data for applying your SQL queries.
  5. In the Table Name field, enter a name for the SQL Table Alias.
    6. NOTE: Xcalar recommends using a name that relates to the originating Xcalar table. For example, use the original Xcalar table name plus sql, such as tablename_sql.
  6. Click APPLY, which does the following:
    • Creates a new SQL Table Alias , which appears in the SQL panel’s Tables and Schemas available to SQL section.
    • Creates a resultant Xcalar table who’s table name and column name are in all capital letters.
    • Replaces the originating Xcalar table that you selected from the Table list, with the new resultant Xcalar table in your active worksheet.
    • Locks the resultant Xcalar table by displaying a lock icon in the top left corner of the resultant Xcalar table.
      • IMPORTANT: Unlocking the resultant Xcalar table by clicking the lock icon, removes your ability to refer to the alias for this Xcalar table in an SQL query.
  7. (Optional) For each Xcalar table that is part or your SQL statement, such as tables for SQL join clauses, stay in the My Extensions panel and repeat steps 3 to 6.

If for any reason you need to revert back to the original Xcalar table, such as you selected the wrong table from the Table list, use one of the options described in Undoing, redoing, and repeating an operation.

Understanding the SQL panel

The Xcalar Design SQL panel contains the following sections for applying SQL query statements on your data and managing your SQL Table Aliases:

  • Xcalar Design SQL code editor. For more information, see Working with SQL query statements in Xcalar tables.
  • Tables and Schemas available to SQL. For more information, see Working with your SQL Table Aliases.

Working with SQL query statements in Xcalar tables

The Xcalar Design SQL code editor is located at the top of the Xcalar Design SQL panel and contains the following features:

  • Syntax highlighting for most of the SQL query statement keywords.
  • Terminating your SQL statement with a semicolon is optional.
  • The SQL query statement keywords are NOT case-sensitive, for example, SELECT is the same as select and FROM is the same as from.
  • SQL verification. Xcalar Design checks your SQL query statements before executing the SQL query. When errors are encountered, such as incorrect SQL syntax or an incorrect table name, an error message appears.
    • TIP: The most common error is applying SQL on a Xcalar table that does not have a SQL Table Alias.

Applying SQL

IMPORTANT: All references to the name of the TABLE in your SQL query statement must use the alias name of the Xcalar table. For example, if your Xcalar table name is transaction and its SQL Table Alias name is transaction_sql, use transaction_sql, such as:
FROM transaction_sql

To apply SQL query statements on your data with the Xcalar Design SQL code editor:

  1. Verify that a SQL Table Alias was created from the active Xcalar table on which you are working.
  2. In the Xcalar Design SQL code editor, enter the SQL query statement.
  3. Click EXECUTE SQL.
  4. (Optional) During the EXECUTE SQL action you can track your SQL query processing operation from the Operations panel. For more information, see Tracking the SQL operation details.

A resultant Xcalar table is created and added to your active worksheet.

Tracking the SQL operation details

You can track the SQL statement's processing details on the Xcalar Design table from the Operations panel.

To track the SQL operation details:

  1. From the sidebar, click the Monitor icon, and then click the Operations icon.
  2. In the Operations panel, click the Show all operations button.
  3. From the Show all operations list, select Execute SQL.

    4. The Operations Details page opens displaying the progress and the processing details of your SQL operation.

Working with your SQL Table Aliases

All SQL Table Aliases for your workbook are listed in the SQL panel’s Tables and Schemas available to SQL section.

Each SQL Table Alias is associated with a schema that contains the column names and data types, such as integer or string, of the originating Xcalar table. You can view the SQL Table Alias's schema by clicking on the name of the SQL Table Alias from the SQL Table Alias list. This action not only displays the column names and data types, it also displays the name of the Xcalar table, and turns the SQL Table Alias name blue.

The Tables and Schemas available to SQL section contains the following functions for working with and managing your SQL Table Aliases:

  • SQL Table Alias list, which lists the available SQL Table Aliases for this worksheet.
  • Table in workbook field, which displays the name of the Xcalar table that was used to create the SQL Table Alias when the SQL Table Alias is selected in the SQL Table Alias list.
  • Tables search tool, which enables you to filter the SQL Table Alias list for alias names specified in the search string.
  • Columns search tool, which enables you to filter the schema list for column names specified in the search string.
  • Column name action, which focuses on the column in your worksheet when you click on column name in the schema.
  • Trash can icon , which is displayed by each SQL Table Alias name and is used to remove an SQL Table Alias from your workbook.

Removing a SQL Table Alias

To remove (drop) a SQL Table Alias from the Tables and Schemas available to SQL section, click the trash can icon  .

IMPORTANT: No confirmation action message appears when you drop a SQL Table Alias. Once removed you cannot undo the drop action by using the undo arrow or any other undo action.
NOTE: Though you may have removed a SQL Table Alias from the SQL panel, all the SQL query operations that you applied to the resultant Xcalar table are still part of your modeling dataflow. If you still require the SQL Table Alias you must recreate the alias as described in Creating a SQL Table Alias.

Xcalar SQL Limitations

The following operations are not supported:

  • A standard deviation on data when applied with other aggregate functions.

  • A frame clause for the following window functions:

    • RANGE UNBOUNDED PRECEDING
    • RANGE BETWEEN CURRENT ROW AND CURRENT ROW
    • [RANGE | ROWS] BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    • [RANGE | ROWS] BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
  • GROUPING SETS and CUBE queries
  • SQL queries that run on less than one table.
  • Multiple SQL query statements for one execution action.
    • NOTE: If you require multiple SQL statements, separate and run each SQL query statement, one after the other.
