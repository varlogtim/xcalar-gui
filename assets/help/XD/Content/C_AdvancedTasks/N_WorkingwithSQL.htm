Working with SQL in Xcalar Design

This section describes using Xcalar SQL in Xcalar Design to manipulate data for analysis.

Understanding Xcalar SQL

Xcalar SQL enables you to run relational database query type processing on your data in Xcalar Design. The Xcalar SQL is based on the traditional SQL ANSI standard.

The Xcalar Design table publishing abstraction SQL extension creates a schema-based alias from your active Xcalar table. It provides support for SQL query statements and operations on structured and semi-structured data by publishing the alias (SQL Table Alias) into the SQL namespace. You can also combine Xcalar Design’s extensive operations with your SQL query statement operations.

The SQL Table Aliases are listed in the Tables and Schemas available to SQL section of the Xcalar Design SQL panel, which provides an easy way to manage your SQL Table Aliases and their schemas.

The SQL code editor of the Xcalar Design SQL panel contains features that simplify the creation and storage of your SQL query statements. You also run your query statements from the code editor, which applies each query simultaneously to the SQL Table Alias, compiles each query, runs the relevant operation, and returns your results in a Xcalar table.

The dataflow graph of your resultant Xcalar table displays all the operations that were executed, including the SQL queries. The SQL Query History displays each SQL query statement applied to your data in more detail. For more information, see Viewing SQL operations from the dataflow graph and Understanding the SQL Query History panel.

Relationship between a Xcalar table and a SQL Table Alias

The SQL Table Alias is an alias created for a Xcalar table that enables you to use SQL query statements that are case-insensitive on a Xcalar table, which uses case-sensitive naming conventions. The alias is associated with a schema that contains the column names and data types, such as integer or string, of the originating Xcalar table.

TIP: When a SQL Table Alias is created the resultant Xcalar table displays the table name and column names in all capital letters.

Creating a SQL Table Alias

You can only create a SQL Table Alias from an active Xcalar table.

To create a SQL Table Alias:

  1. Verify that the Xcalar table that contains the data for applying your SQL queries is active.
  2. From the sidebar, click the Extensions icon.

    3. The My Extensions panel opens.

  3. Locate the SQL extension and then select Create SQL Table Alias.

    4. The SQL:Create SQL Table Alias section opens.

  4. From the Table list, select the Xcalar table that contains the data for applying your SQL queries.
    5. TIP: If your table does not appear in the Table list, verify that your Xcalar table is active and is in the worksheet in which you are currently working.
  5. In the Table Name field, enter a name for the SQL Table Alias.
    6. NOTE: Xcalar recommends using a name that relates to the originating Xcalar table. For example, use the original Xcalar table name plus sql, such as tablename_sql. For more information on SQL Table Alias naming conventions, see SQL Table Alias table naming conventions.
  6. Before proceeding, verify that the name you entered in the Table Name field is original.
    7. IMPORTANT: A SQL Table Alias can be overwritten if you enter the exact same name of an existing SQL Table Alias. No warning message appears after you click APPLY and the existing SQL Table Alias will no longer be available.
  7. Click APPLY, which does the following:
    • Creates a new SQL Table Alias, which appears in the SQL panel’s Tables and Schemas available to SQL section.
    • Creates a resultant Xcalar table whose table name and column names are capitalized.
    • Replaces the originating Xcalar table that you selected from the Table list with the resultant Xcalar table in your active worksheet.
    • Moves the originating Xcalar table that you selected from the Table list to the Temporary Tables list.
    • Locks the resultant Xcalar table and displays a lock icon in the top left corner.
      • IMPORTANT: Whereas, unlocking the resultant Xcalar table by clicking the lock icon does not prevent you from referring to the alias, dropping (removing) the originating Xcalar table also drops the SQL Table Alias. Xcalar recommends that you do not unlock this table since this prevents the table from being dropped accidentally.
  8. (Optional) For each Xcalar table that is part of your SQL statement, such as tables required for SQL join clauses, stay in the My Extensions panel and repeat steps 2 to 7.
    9. NOTE: If you need to revert to the originating Xcalar table, for instance you selected the wrong table from the Table list, use one of the options described in Undoing, redoing, and repeating an operation.
    IMPORTANT: When you undo the Create SQL Table Alias operation, the SQL Table Alias is not removed. It is listed in the SQL panel’s Tables and Schemas available to SQL section and is available as a table alias for SQL queries. The resultant Xcalar table created from the Create SQL Table Alias operation is also not removed but is moved to the Temporary Tables list.

Displaying the Xcalar Design SQL panel

To display the SQL panel, click the SQL  icon.

Understanding the SQL panel

The Xcalar Design SQL panel contains the following sections for applying SQL query statements on your data and managing your SQL Table Aliases:

  • Xcalar Design SQL code editor. For more information, see Working with SQL query statements in Xcalar tables.
  • Tables and Schemas available to SQL. For more information, see Working with your SQL Table Aliases.

Working with SQL query statements in Xcalar tables

The Xcalar Design SQL code editor is located at the top of the Xcalar Design SQL panel and contains the following features:

  • Syntax highlighting for most of the SQL query statement keywords.
  • Autocomplete suggestion and insertion assistance.

    • When entering your query statement in the Xcalar SQL code editor, autocomplete suggestions appear, such as SQL keywords and SQL Table Alias names and column names. Select and accept the correct suggestion by pressing the tab or enter key.

  • Case-insensitive keywords.

    • The Xcalar SQL query statement keywords are not case-sensitive. For example, SELECT is the same as select and FROM is the same as from.

  • SQL keyboard shortcuts.

    • Xcalar provides SQL keyboard shortcuts. For example, you can either manually add the double-hyphen (--) comment delimiter before a line comment , such as --example comment, or depending on your operating system, use the Xcalar SQL Keyboard shortcut Ctrl+ - or control+ -. For more information on the Xcalar SQL Keyboard shortcuts, see Xcalar SQL keyboard shortcuts.

  • Optional semicolon termination for a single SQL statement.

    • When applying a single SQL query statement, terminating the SQL query statement with a semicolon is optional.

    TIP: For code snippets containing multiple SQL statements that are not terminated with a semicolon, as they are intended to be separately applied, you must highlight the single statement that is to be applied before clicking EXECUTE SQL. If you do not highlight the single statement, all queries are handled as one single SQL query statement.
  • Multiple SQL query statement execution.

    • You can run multiple SQL queries simultaneously with one EXECUTE SQL click. For a multiple SQL query statement execution, you must end each SQL query statement with a semicolon.

  • Stores multiple SQL queries in a code snippet.

    • For more information, see Storing SQL query statements in a code snippet.

  • Automatically saves query entries.

    • You can enter one or multiple SQL query statements in the Xcalar Design SQL code editor, which remain until you remove them, even after you log out of Xcalar Design

  • SQL verification.

    • Xcalar Design verifies your SQL query statements. When errors are encountered, such as incorrect SQL syntax or an incorrect table name, an error message appears. Error messages are saved and viewable from the SQL History panel. For more information, see Understanding the SQL Query History panel.

    TIP: The most common error is trying to apply SQL on a Xcalar table that does not have a SQL Table Alias.

Applying SQL

IMPORTANT: All table names in your SQL query statement must use the alias name of the active Xcalar table. For example, if your active Xcalar table name is transaction and its SQL Table Alias name is transaction_sql, use transaction_sql in the SQL statement, as shown in the following example:

SELECT * FROM transaction_sql

To apply SQL:

  1. Verify that a SQL Table Alias was created from the active Xcalar table in which you are working.
  2. (Optional) From the code snippet list, create a new code snippet to store your SQL queries. For more information, see Storing SQL query statements in a code snippet.
  3. In the Xcalar Design SQL code editor, do one of the following options:
    • To apply multiple SQL query statements simultaneously to your SQL Table Alias, enter each SQL query statement followed by a semicolon.
    • To apply a single SQL query statement to your SQL Table alias, enter one or more SQL query statements with or without a semicolon.
      • NOTE: This option requires that you highlight the single SQL query before clicking EXECUTE SQL.
  4. Click EXECUTE SQL, which does the following:
    • Creates a resultant Xcalar table in the active worksheet. To apply SQL on this resultant Xcalar table, you must create a new SQL Table Alias from this resultant Xcalar table.
    • Keeps the active Xcalar table in the Active table list and worksheet. You can continue to apply SQL on this active Xcalar table using its SQL Table Alias.
    • Creates a temporary table for each SQL query operation.
  5. (Optional) Track your SQL query processing operations from the Operations panel. For more information, see Tracking the SQL operation details.
  6. (Optional) View and verify your SQL query history from the SQL Query History panel. For more information, see Understanding the SQL Query History panel.

Storing SQL query statements in a code snippet

The SQL code editor automatically saves your SQL query statements, even after you log out of Xcalar Design.

You can also store SQL query statements for reuse in a code snippet.

To store SQL query statements in a code snippet:

  1. At the top of the SQL code editor, in the code snippet search field, click the down arrow.

    2. The code snippet list appears.

  2. Select New Snippet.
  3. In the Enter snippet name field, enter a name for the snippet.

    4. The new snippet appears in the code snippet search field.

  4. In the SQL code editor, enter your SQL query or queries.

Managing SQL code snippets

The following management tasks can be performed from the SQL code snippet list:

  • Creating a new code snippet.

    • For more information, see Storing SQL query statements in a code snippet.

  • Renaming a code snippet.
    1. At the top of the SQL code editor, in the code snippet search field, click the down arrow.

      2. The code snippet list appears.

    2. From the code snippet list, click the pencil icon at the side of the code snippet's name that is to be renamed.
    3. In the code snippet's name field, replace the code snippet's name with a new snippet name.
    4. Press Enter.
  • Deleting a code snippet.
    1. At the top of the SQL code editor, in the code snippet search field, click the down arrow.

      2. The code snippet list appears.

    2. From the code snippet list, click the trash can icon at the side of the code snippet's name that is to be deleted.
    IMPORTANT: No confirmation message appears when you delete a code snippet. Once deleted you cannot undo the delete action by clicking the undo icon or any other undo action.

Understanding the Default Snippet

The Default Snippet enables you to quickly apply SQL queries on your SQL Table aliases. The SQL code editor automatically saves any SQL code that you enter in the Default Snippet, even after you log out of Xcalar Design.

You can change the name of the Default Snippet by entering a new name in the code snippet search field and pressing Enter.

RECOMMENDATION: The Default Snippet name is permanently changed to the new snippet name and the Default Snippet no longer appears in the code snippet list. To retain the Default Snippet, copy the SQL code that you have entered in the Default Snippet to a new code snippet.
NOTE: To ensure that there is always a code snippet available for you to create new SQL queries, the Default Snippet will reappear in the code snippet list if all the code snippets are deleted.

Tracking the SQL operation details

You can track the processing details of your SQL query statement from the Xcalar Design Operations panel.

To track the SQL operation details:

  1. From the sidebar, click the Monitor icon, and then click the Operations icon.
  2. In the Operations panel, click the Show all operations button.

    3. The Show all operations list opens to the last operation entry at the bottom of the list.

  3. From the Show all operations list, select Execute SQL.

    4. The Operations Details page opens, displaying the progress and the processing details of your SQL query.

  4. (Optional) To view the resultant Xcalar table, click VIEW OUTPUT.
NOTE: When applying multiple SQL queries, multiple Execute SQL operations will appear in the Show all operations list.

Understanding the SQL Query History panel

The SQL Query History panel displays in detail the SQL query statements that were applied to your data.

To open the SQL Query History panel:

  1. From the sidebar, click the Monitor icon.
  2. From the Monitor icon's sublist, click the SQL Query History icon.
TIP: The Query History page lists your queries from the most recent operation, which is based on the Start Time column values. You can also sort the Query History from the Duration and Status column values, by clicking in their column heading. Where the down and up arrows indicate the sort order.

The following table describes the entry fields of the Query History page:

Field

Description

Status

Lists the final processing execution state of the query statement:

  • Canceled - the query operation did not complete because the query was deliberately stopped by the user. For example, the user pressed the cancel shortcut key.
  • Compiling - the query operation is being compiled.
  • Done - the query operation successfully completed processing.
  • Failed - the query operation did not complete due to incorrect or incomplete SQL syntax. For more details, open the SQL ERROR window from the View Results column.
  • Interrupted - the query operation did not complete due to either a network failure or user interference. For example, the user refreshed the browser window.
  • Running - the query operation is processing the query on the data.

Query

Displays the query that was applied to the data.

To fully display the query value, in the Query field, click on the query, which opens a SQL Query window.

Start Time

Displays the exact time the query was run using the mm-dd-yyyy hr:mins:secs date format. For example, 7-31-2018 3:35:44.

Duration

Displays how long the query took to complete its run.

If the query was interrupted or canceled, N/A is displayed, which represents not applicable.

View Results

Displays the results of the query:

  • If successful, the field is populated with a link to the resultant Xcalar table.
  • If unsuccessful, the field is populated with a link to the SQL error message.
  • If the Xcalar table is dropped, the field is populated with an error message that states the table has been dropped and therefore the link to the table no longer exists.

Working with your SQL Table Aliases

All SQL Table Aliases for your workbook are listed in the SQL panel’s Tables and Schemas available to SQL section.

Each SQL Table Alias is associated with a schema that contains the column names and data types, such as integer or string, of the originating Xcalar table. You can view the schema of a SQL Table Alias by clicking on the name of the SQL Table Alias from the SQL Table Alias list. This action not only displays the column names and data types, it also turns the SQL Table Alias name blue, displays the name of the Xcalar table, and brings the Xcalar table into focus.

The Tables and Schemas available to SQL section contains the following functions for working with and managing your SQL Table Aliases:

  • SQL Table Alias list, which lists the available SQL Table Aliases for this worksheet.
  • Table in workbook field, which displays the name of the Xcalar table that was used to create the SQL Table Alias.
  • Tables search tool, which enables you to filter the SQL Table Alias list for alias names specified in the search string. Results automatically appear as you type.
  • Columns search tool, which enables you to filter the schema list for column names specified in the search string. Results automatically appear as you type.
  • Column name action, which focuses on the column in your worksheet when you click on a column name in the schema.
  • Trash can icon, which is displayed by each SQL Table Alias name and is used to remove an SQL Table Alias.

Removing a SQL Table Alias

To remove (drop) a SQL Table Alias from the Tables and Schemas available to SQL section, click the trash can  icon.

IMPORTANT: No confirmation message appears when you drop a SQL Table Alias. Once removed you cannot undo the drop action by using the undo icon or any other undo action.
NOTE: After you have removed a SQL Table Alias from the SQL panel, all the SQL query operations that you applied to the resultant Xcalar table remain in the table's dataflow. If you still require the SQL Table Alias you must recreate the alias as described in Creating a SQL Table Alias.

SQL Table Alias table naming conventions

The following list describes the naming conventions:
  • Characters must be A-Z, a-z, 0-9, hyphen (-), underscore (_), or space.
  • First character must be a letter.
  • Length must not exceed 255 characters.

When the SQL Table Alias is generated from the Xcalar table, its name is automatically processed into uppercase by Xcalar and made case-insensitive.

IMPORTANT: A SQL Table Alias can be overwritten. No warning message appears when you create a SQL Table Alias that uses the exact same name as an existing SQL Table Alias. Because the existing alias is overwritten, it is no longer part of its original Xcalar table.

Xcalar SQL Limitations

The following operations are not supported:

  • A standard deviation on data when applied with other aggregate functions.
  • SQL queries that run on less than one table.

Window function restrictions:

The following window functions are supported without the OVER clause:

  • Values - FIRST_VALUE(), LAST_VALUE(), LAG(), LEAD()
  • Ranking - CUME_DIST(), DENSE_RANK(), PERCENT_RANK(), ROW_NUMBER()

IMPORTANT: Ranking functions depend on sort ordering specified by the ORDER BY clause, which is part of the OVER clause. If you do not include the OVER clause when using ORDER BY, the results are non-deterministic.

Frame clause restrictions:

In the frame clause of a window function, you must specify where the frame starts and ends.

The Xcalar SQL default frame clause includes all the values from the start of the partition through to the last row of the partition:

[RANGE | ROWS] BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING

Where:

  • UNBOUNDED PRECEDING is where the frame starts with the first row of the partition.
  • UNBOUNDED FOLLOWING is where the frame ends with the last row of the partition.

The following frame clauses are not supported:

  • RANGE UNBOUNDED PRECEDING
  • [RANGE | ROWS] BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  • RANGE BETWEEN CURRENT ROW AND CURRENT ROW
