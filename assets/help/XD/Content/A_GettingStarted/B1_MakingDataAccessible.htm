Making source data accessible

Importing a data source means telling Xcalar Design to derive metadata from one or multiple files containing raw data. These files must be stored at a location accessible to all Xcalar nodes for Xcalar Design to import them.

IMPORTANT: After you import data from a file to Xcalar, changing the path to the file renders your data inaccessible after a cluster restart. For example, if you rename the source data file or the directory containing the file, and then restart the cluster, Xcalar Design can no longer locate the data source when trying to import the data. This failure means that you cannot use the workbook containing tables created from the data. Therefore, name carefully each component of the path to the source data file so that the path does not require any changes after the import.

This topic describes the steps for ensuring that the source data files are accessible for Xcalar to import. Some of these steps require configuring the storage system where the source data files reside. If you do not have the appropriate access privileges, contact your system administrator to set up the source data files for you.

Understanding data targets

Xcalar can import data from various storage platforms, be they in the cloud or on-premises. When you import data, you must provide information about the storage location and the method used by your Xcalar instance to access the location. A data target is the specifications of the storage location and the access method. At the time of importing data, you select a path on a data target, and Xcalar Design will process the file or files in the specified path according to the specifications. Xcalar provides predefined data targets, but you can create additional ones.

For example, your data target can be a Microsoft Azure Storage Account, an Amazon Simple Storage Service (S3) account, or an NFS-mounted file system on your Xcalar cluster. Depending on the target type, Xcalar Design accesses the files by using the authentication method described in the target.

For information about how the data target type affects the way Xcalar Design accesses the data source, see Data target types.

Predefined data targets

The following table describes the data targets that ship with Xcalar Design:

Data target Description Data target type

Default shared root

 The file system mounted on the / directory of each Xcalar cluster node. Shared Storage Mount

Preconfigured Azure Storage Account

 The Azure Storage Account that Xcalar creates when you deploy Xcalar in Microsoft Azure Marketplace. Azure Blob Storage Account with Environmental Credentials

Preconfigured Google Cloud Storage Account

 The Google Cloud Storage account that Xcalar creates for you when you deploy Xcalar on Google Cloud Platform. Google Cloud Storage Account with Environmental Credentials

Preconfigured S3 Account

 The S3 account that Xcalar creates for you when you deploy Xcalar on Amazon Web Services. S3 Account with Environmental Credentials
NOTE: You cannot delete a predefined data target.

Data target types

You can store source data files at any predefined data target. In addition, you can create data targets. When creating a data target, specify its type. The data target type determines what information you must provide for authentication when your Xcalar instance tries to access the data source.

NOTE: The phrase environmental credentials in the data type name means that the credentials for accessing the data source are set up outside of the data target creation process.

The following table describes the data target types.

Data target type Description
Azure Blob Storage Account with Environmental Credentials An Azure Storage account accessible with the account information stored in the environment variables, which are AZURE_STORAGE_ACCOUNT and AZURE_STORAGE_ACCESS_KEY.
Azure Blob Storage Account with SAS Token An Azure Storage account that you access with a shared access signature (SAS) token.
Google Cloud Storage Account with Environmental Credentials A Google Cloud account for which you set up the account credentials by following the instructions from Google. For information about Google Cloud account authentication, see https://cloud.google.com/storage/docs/authentication.
HttpFs with Kerberos An HDFS file system that uses Kerberos for authentication and is accessed through the HttpFS protocol.
HttpFs An HDFS file system that is accessed through the HttpFS protocol.
Secured MapR Cluster A MapR cluster that you access with credentials created for the MapR cluster.
Generated A dataset created by an import UDF. The data exists in memory only, not in a storage system. This data target type is not common; it is used mainly by advanced users for testing purposes.
S3 Account with Environmental Credentials An S3 account that you access with the Boto 3 credentials. These credentials are contained in an AWS configuration file on your cluster.
S3 Account with Specified Credentials An S3 account that you access with the credentials created for the S3 account. You must specify the credentials for the S3 account when you create the target.
Shared Storage Mount A mount point with the same directory name across all nodes, to which the same file system is attached. For example, for each node, you can NFS-mount on /mnt/data a file system named airlinesData on a storage system.
Single Node Filesystem A file system that exists only on a single node. When importing from a target of this type, you specify the node number followed by the path of the source data file, and Xcalar imports only from the specified node.
Symmetric Pre-sharded Filesystem

A directory with the same name on each node, which contains different files you created by sharding a large file.

EXAMPLE: You can split a data source file named airlines.csv into airlines1.csv, airlines2.csv, airlines3.csv, and airlines4.csv. Then save two of these files to /airlinesData on node 0 and the other two files to /airlinesData on node 1. After you create a data target of the Symmetric Pre-sharded Filesystem type for /airlinesData, you can import all four files from this target.
WebHDFS with Kerberos An HDFS file system that uses Kerberos for authentication and is accessed through the WebHDFS protocol.
WebHDFS An HDFS file system that is accessed through the WebHDFS protocol.

Before creating a data target

For the following data target types, prepare the data target by performing additional steps provided in the Create Data Target window:

  • HttpsFs with Kerberos
  • HttpsFs
  • WebHDFS with Kerberos
  • WebHDFS

For other data target types, generate or gather information in advance as described in this section.

Azure Storage Account with SAS Token

Follow these steps to generate a SAS token, which is required when you create the data target:

  1. Use a web browser to go to your Azure Storage accounts management web page.
  2. Click your storage account name.
  3. Navigate to the shared access signature (SAS) page.

  4. Set the SAS parameters as follows to generate the SAS token. Leave all other parameters not listed in the following table at their defaults.

    Parameter Setting
    Start and expiry date/time Start time and end time for the time period when your Xcalar cluster can access the data
    Allowed protocols HTTPS only

  5. After the SAS token is generated, copy the token (not the signature) to your computer's clipboard. You will need to paste the token when you create the data target.

    The following is an example of a SAS token with a Start Time set to 2017-09-18 00:00;00 and End Time set to 2017-12-01 00:00:00:

    ?sv=2017-04-17&ss=b&srt=co&sp=rl&se=2017-12-01T00:00:00Z&st=2017-09-18T00:00:00Z&spr=https&sig=xO0OwmPaIe27S0jy%2Fqr%2FbRs6WXj3%2BP84fGr4%2Bh%2BE5Qs%3D

Shared Storage Mount

If your source data files are in shared storage visible to all nodes (for example, through a Network File System (NFS) in a network attached storage (NAS) environment), verify that the directory structure for accessing the shared storage is identical across all nodes.

For example, if your data resides in an NFS file system, all nodes must mount the same file system using the same path name as the mount point. That is, the same file system must be mounted on all nodes, and the mount point on all nodes must be the same (for example, /mnt/datasets). Similarly, if the source data files are local files, they must be stored under the same name and they must have identical contents. For example, to import a local file named /datasets/file1, Xcalar Design must be able to find /datasets/file1 on each node, and /datasets/file1 must contain the same contents across all nodes.

The following list describes the possible scenarios where importing a shared file system fails:

  • The nodes use different mount points for the same NFS file system. For example, some nodes mount the file system as /mnt/dataset1 and some nodes mount the file system as /mnt/dataset2.
  • All nodes have the same mount point (/mnt/datasets) but it points to different file systems. For example, /mnt/datasets on some nodes point to a file system on host1 and the same mount point on other nodes point to a file system on host2.

S3 Account with Specified Credentials

IMPORTANT: Xcalar recommends that you use the S3 Account with Environmental Credentials as the target type if you import from an S3 account. This is because the information entered to create a target of the S3 Account with Specified Credentials type is not secured.

Obtain the following information from AWS when you create the data target:

  • Access key
  • Secret access key
  • Region name
  • Endpoints

For information about access keys, see http://docs.aws.amazon.com/general/latest/gr/aws-sec-cred-types.html. For information about region and endpoints, see http://docs.aws.amazon.com/general/latest/gr/rande.html.

S3 Account with Environmental Credentials

The S3 instance needs to be configured for S3 access. This is because Xcalar connects not only to your local S3 bucket, but also to any other S3 accounts that you have access to.

To configure S3, establish an SSH connection with each Xcalar cluster node. Then follow these steps:

  1. Make sure that you have installed the Amazon Web Services (AWS) Command Line Interface (CLI). If not, follow the instructions to install it at https://aws.amazon.com/cli/.
  2. To set up your credentials and default region, enter the following command:

    3. aws configure

    The command displays several prompts for you to enter the access key, default region name, and so on. After you finish responding to the prompts, configuration files are generated in the correct locations for you.

For detailed information and other ways to configure S3, visit the following website:

http://boto3.readthedocs.io/en/latest/guide/configuration.html#configuring-credentials

Secured MapR Cluster

If the data source is on a MapR cluster, follow these steps to enable authorized MapR cluster users to access the data from Xcalar Design:

  1. Verify that the Xcalar users who want to access data on a MapR cluster have their usernames and passwords for the MapR cluster.
  2. For each Xcalar cluster node, follow these steps to set up the MapR client:

    1. Install the appropriate MapR client package from the MapR website. For example, install the following package from MapR:

      mapr-client-5.2.0.39122.GA-1.x86_64.rpm

      Make sure that the client package version matches your MapR cluster version.

    2. Enter the following command, which uses the MapR configure script, on the Xcalar cluster node to configure the MapR client against the MapR cluster:

      /opt/mapr/server/configure.sh -c -C mapr-cldb-node -secure -N my-mapr-cluster

      In the actual command, substitute your MapR cluster name for my-mapr-cluster.

    3. Enter the following command to copy the cluster ssl_truststore to the client node:

      scp admin@mapr-cldb-node:/opt/mapr/conf/ssl_truststore /opt/mapr/conf

  3. (Optional) Follow these steps to verify that each user can log in to the MapR cluster:

    1. Enter the maprlogin. The following example is for a user named myuser:

      maprlogin password -user myuser

      In the actual command, substitute the MapR user's username for myuser.

    2. Enter the MapR user's password for the MapR cluster.
    3. If the login is successful, you can be sure that the user can access the data on the MapR cluster from Xcalar Design.

    4. Enter the following command to log out of the MapR cluster:

      maprlogin logout

Creating a data target

Follow these steps to create a data target:

  1. Click the Datasets icon if the Data Targets icon is not already shown. The following screenshot shows the location of the Data Targets icon.

  2. In the Data Targets panel, click any data target icon to display information about the data target. If your desired data target does not exist, click the Create Data Target icon to start creating a data target.
  3. In the Create Data Target window, type the name of the data target. For example, you can create a data target named airlinesInfo.
  4. Select the appropriate data target type. Depending on the type, you must enter additional information in the fields displayed. For example, if the type is Shared Storage Mount, enter the mount point on each node for the shared storage.
  5. Click ADD.
NOTE: After you create a data target, you cannot edit its information, such as the name, the mount point for a Shared Storage Mount target, and so on. To change the information, you must delete the target and create a target with the desired information.
IMPORTANT: Before deleting a data target, make sure that no workbooks on the cluster depend on datasets that import from the data target. After a cluster restart, the owner of a workbook containing data from the deleted target will not be able to activate the workbook.
