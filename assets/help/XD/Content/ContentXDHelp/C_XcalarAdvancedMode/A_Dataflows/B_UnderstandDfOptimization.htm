Understanding dataflow execution

You can execute a dataflow either with or without optimization. The dataflow, however, must meet the requirements described in Requirements for an optimized execution of a dataflow for you to execute it with optimization.

What an optimized execution does

The following list describes what happens when you execute a dataflow with optimization:

  • If the dataflow's first node is a DATASET node, an optimized execution re-imports data from the data source and all subsequent operations use the latest data from that import. Without optimization, the execution uses the data imported when the DATASET node was first created.
  • Xcalar uses an optimizer during the execution of the dataflow, enabling all operations to produce results in the most efficient manner.
  • An optimized execution avoids the risk of out-of-memory problems. Xcalar automatically performs serialization if necessary when executing an optimized dataflow.

After you execute a dataflow with optimization, you cannot re-execute it with optimization. If you try to do so, Xcalar notifies you that an optimized execution already took place.

Factors for deciding whether to execute a dataflow with optimization

Execute a dataflow with optimization if the following conditions are true:

  • You have finalized your dataflow. You are more interested in the final result of the dataflow than the result sets of individual operations, which are intermediate steps in the algorithm.
  • You want to operationalize the dataflow. For example, you want to execute the dataflow regularly on a schedule through the SDK.

Execute a dataflow without optimization when you are developing an algorithm. The execution steps through the operations, creating a result set for each. Consequently, you can examine the result sets to determine whether the operations are configured correctly, the operations are properly ordered, and so on.

Requirements for an optimized execution of a dataflow

The following list describes the requirements for executing a dataflow with optimization:

  • The dataflow must not contain a terminal node that is not Link Out Optimized or Export Optimized. For example, if the filter operation exists as one of the terminal nodes of the dataflow, you cannot execute the dataflow with optimization.
  • The dataflow must not contain multiple Link Out Optimized nodes.
  • The dataflow must not contain both Link Out Optimized and Export Optimized nodes. However, it can have multiple Export Optimized nodes.

Where to start the execution of an optimized dataflow

You can start an optimized execution of a dataflow in the following ways:

  • Execute all operations in a dataflow with optimization.
  • Execute a Link Out Optimized node or an Export Optimized node.

The optimized execution of the dataflow generates an image of the dataflow after optimization. You can inspect the configuration of each node and statistics from the execution.

NOTE:The nodes in the optimized dataflow, as shown in the image, might not have a one-to-one relationship with the nodes in the dataflow. Optimizing a dataflow is analogous to using the -O3 flag in the GNU Compiler Collection (GCC): The output assembly code generated by the compiler need not have a one-to-one relationship with the source code.
