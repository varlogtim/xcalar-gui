Understanding dataflow execution

You can execute a dataflow with or without optimization. The dataflow, however, must meet the requirements described in Requirements for an optimized execution for you to execute it with optimization.

What an optimized execution does

The following list describes what happens when you execute a dataflow with optimization:

  • If the dataflow's first Node is a Dataset Node, an optimized execution re-imports data from the data source and all subsequent operations use the latest data from that import. Xcalar only re-imports the data that is required to execute the dataflow. Unused fields are not imported and therefore consume no memory. Without optimization, the execution uses the data imported when the Dataset Node was first created.
  • Xcalar uses an optimizer during the execution of the dataflow, enabling all operations to produce results in the most efficient manner.
  • An optimized execution avoids the risk of out-of-memory problems. Xcalar automatically performs serialization if necessary when executing an optimized dataflow.

After you execute a dataflow with optimization, you cannot re-execute it with optimization. If you try to do so, Xcalar notifies you that an optimized execution already took place.

Factors for deciding whether to execute a dataflow with optimization

Execute a dataflow with optimization if the following conditions are true:

  • You have finalized your dataflow. You are more interested in the final result of the dataflow than the result sets of individual operations, which are intermediate steps in the algorithm.
  • You want to operationalize the dataflow. For example, you want to execute the dataflow regularly on a schedule through the Xcalar SDK.

Execute a dataflow without optimization when you are developing an algorithm. Without optimization, the execution steps through the operations, creating a result set for each. Consequently, you can examine the result sets to determine whether the operations are configured correctly, the operations are properly ordered, and so on.

Requirements for an optimized execution

For a dataflow to be executed with optimization, it must meet all of the following requirements:

  • The dataflow must not contain sub-dataflows. For example, in the following screenshot, Dataflow 9 contains two sub-dataflows.  You cannot perform an optimized execution on Dataflow 9. You can, however, execute each sub-dataflow with optimization.

  • The dataflow must not contain a terminal Node that is not Link Out Optimized or Export Optimized. For example, in the following screenshot, the dataflow cannot be executed with optimization because one of the terminal Nodes is a filter operation.

  • The dataflow must not contain multiple Link Out Optimized Nodes. For example, in the following screenshot, the dataflow cannot be executed with optimization because it has two Link Out Optimized Nodes.

  • The dataflow must not contain both Link Out Optimized and Export Optimized Nodes. For example, in the following screenshot, the dataflow cannot be executed with optimization because it has a mix of Link Out Optimized and Export Optimized Nodes.

    However, the dataflow can be executed with optimization if it contains multiple Export Optimized Nodes.

Where to start an optimized execution

You can start an optimized execution of a dataflow in the following ways:

  • Execute all operations in a dataflow with optimization.
  • Execute a Link Out Optimized Node or an Export Optimized Node.

For information about how to start an optimized execution, see Executing a dataflow.

The optimized execution of the dataflow generates an optimized dataflow in a separate dataflow tab. You can inspect the configuration of each Node and statistics from the execution.

NOTE:The Nodes in the optimized dataflow, as shown in the newly created dataflow tab, might not have a one-to-one relationship with the Nodes in the dataflow. Optimizing a dataflow is analogous to using the -O3 flag in the GNU Compiler Collection (GCC): The output assembly code generated by the compiler need not have a one-to-one relationship with the source code.
