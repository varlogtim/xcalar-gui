Making source data accessible

Importing a data source means telling Xcalar Design to derive metadata from one or multiple files containing raw data. These files must be stored at a location accessible to all Xcalar nodes for Xcalar Design to import them.

This topic describes the steps for ensuring that the source data files are accessible for Xcalar to import. Some of these steps require configuring the storage system where the source data files reside. If you do not have the appropriate access privileges, contact your system administrator.

NOTE: If you are a Xcalar user, read this topic to understand data targets. Most of the information in this topic, however, is for the Xcalar cluster administrator, who has the privilege to create or edit data targets.

Understanding data targets

Xcalar can import data from various storage platforms, be they on cloud or on on-premises platforms. When you import data, you must provide information about the storage location and the method used by your Xcalar instance to access the location. A data target contains the specifications of the storage location and the access method. At the time of importing data, you select a path on a data target, and Xcalar Design will process the file or files in the specified path according to the specifications. Xcalar provides a predefined data target, but you can create additional ones.

For example, your data target can be a Microsoft Azure Storage Account, an Amazon Simple Storage Service (S3) account, or an NFS file system mounted on your Xcalar cluster. Depending on the target type, Xcalar Design accesses the files by using the authentication method described in the target.

For information about how the data target type affects the way Xcalar Design accesses the data source, see Data target types.

NOTE: You can create or delete data targets only if you are the cluster administrator. Contact the cluster administrator if your data source is not located on any of the existing data targets.

Understanding the default data target

The Default Shared Root target is provided by default. It is the file system mounted on the / directory of each Xcalar cluster node. Its data target type is Shared File System.

NOTE: You cannot delete or rename a predefined data target.

Data target types

To access source data stored at a location other than a shared file system, create a data target. When creating a data target, specify its type. The data target type determines what information you must provide for authentication when your Xcalar instance tries to access the data source.

NOTE: The phrase environmental credentials in the data type name means that the credentials for accessing the data source are set up outside of the data target creation process.

The following table describes the data target types.

Data target type Description
Azure Blob Storage Account with Environmental Credentials An Azure Storage account accessible with the account information stored in the environment variables, which are AZURE_STORAGE_ACCOUNT and AZURE_STORAGE_ACCESS_KEY.
Azure Blob Storage Account with SAS Token An Azure Storage account that you access with a shared access signature (SAS) token.

Custom Target

An advanced feature that requires a thorough understanding of how Xcalar uses UDFs to import various kinds of data sources. Contact Xcalar Professional Services for information about this target type.

Database Connector Target

A SQL database.

Google Cloud Storage Account with Environmental Credentials A Google Cloud account for which you set up the account credentials by following the instructions from Google. For information about Google Cloud account authentication, see https://cloud.google.com/storage/docs/authentication.
HttpFs with Kerberos An HDFS file system that uses Kerberos for authentication and is accessed through the HttpFS protocol.
HttpFs An HDFS file system that is accessed through the HttpFS protocol.
Secured MapR Cluster Account An account on a MapR cluster that a user can access with credentials created for that user on the MapR cluster.

Secured MapR Cluster Account Impersonation

A MapR cluster that allows a Xcalar user to access by using an authorized MapR cluster account.

Generated A dataset created by an import UDF. The data exists in memory only, not in a storage system. This data target type is not common; it is used mainly by advanced users for testing purposes.
Parquet Dataset A Parquet format data source.
S3 Account with Environmental Credentials An S3 account that you access with the Boto 3 credentials. These credentials are contained in an AWS configuration file on your cluster.
S3 Account with Specified Credentials An S3 account that you access with the credentials created for the S3 account. You must specify the credentials for the S3 account when you create the target.
Shared File System A mount point with the same directory name across all nodes, to which the same file system is attached. For example, for each node, you can mount on /mnt/data an NFS file system named airlinesData that resides on a storage system.
Single Node Filesystem A file system that exists only on a single node. When importing from a target of this type, you specify the node number followed by the path of the source data file, and Xcalar imports only from the specified node.
Symmetric Pre-sharded Filesystem

A directory with the same name on each node, which contains different files you created by sharding a large file.

EXAMPLE: You can split a data source file named airlines.csv into airlines1.csv, airlines2.csv, airlines3.csv, and airlines4.csv. Then save two of these files to /airlinesData on node 0 and the other two files to /airlinesData on node 1. After you create a data target of the Symmetric Pre-sharded Filesystem type for /airlinesData, you can import all four files from this target.
WebHDFS with Kerberos An HDFS file system that uses Kerberos for authentication and is accessed through the WebHDFS protocol.
WebHDFS An HDFS file system that is accessed through the WebHDFS protocol.

Creating a data target (administrator only)

IMPORTANT: Depending on the data target type, you need to perform steps before creating a data target. For information about the required steps, see the Xcalar Administration Guide.

You can create a data target whether you are in SQL Mode or Advanced Mode. To create a data target, follow these steps:

  1. Click the Datasets icon if the Data Targets icon is not already shown. The following screenshot taken in SQL Mode shows the location of the Data Targets icon.

  2. In the Data Targets panel, click any data target icon to display information about the data target. If your desired data target does not exist, click the Create Data Target icon to start creating a data target.
  3. In the Create Data Target window, enter the name of the data target. For example, you can create a data target named airlinesInfo.
  4. Select the appropriate data target type. Depending on the type, you must enter additional information in the fields displayed. For example, if the type is Shared File System, enter the mount point on each node for the shared storage.
  5. Click ADD.
NOTE: After you create a data target, you cannot edit its information, such as the name, the mount point for a Shared File System target, and so on. To change the information, you must delete the target and create a target with the desired information.
IMPORTANT: Before deleting a data target, make sure that no workbooks on the cluster depend on the data imported from the data target. After a cluster restart, you cannot activate a workbook containing data from the deleted target.
