Understanding the connector types

This section lists the connector types that are supported by Xcalar, which are based on the source data's storage repository type and access authentication method.

Accessing source data stored in a repository other than the Xcalar default shared file system requires a Xcalar Connector, which contains metadata about your source data's format, storage location and repository type, and access authentication method.

NOTE: The text Environmental Credentials in a Connector Type name indicates that the credentials for accessing the source data are configured from environment variables outside of the connector creation process.

The following table describes the Xcalar connector types:

Table 1 Connector types

Source data storage

 Connector type Description

Microsoft Cloud storage

 Azure Blob Storage Account with Environmental Credentials

An Azure Storage account that authorizes access from an Azure storage account authentication file and either a shared access signature (SAS) token or an access key, whose locations are set in the following environment variables:

  • AZURE_STORAGE_ACCOUNT
  • AZURE_STORAGE_KEY
  • AZURE_STORAGE_SAS_TOKEN

Microsoft Cloud storage

 Azure Blob Storage Account with SAS Token An Azure Storage account that requires access with a shared access signature (SAS) token.

 

Custom

This advanced feature requires a complete understanding of how Xcalar uses user-defined functions (UDFs) to import various kinds of data sources. For more information about creating this connector type, contact Xcalar Professional Services.

Relational database

Database

A SQL database, data warehouse, or data lake, such as the following:

  • Apache data warehouse
  • IBM Db2
  • MySQL and MS SQL Server
  • Oracle Database
  • PostgreSQL
  • Teradata

For more information about creating this connector type, contact Xcalar Professional Services.

Pro-grammatically generated in memory

 Generated A dataset of fake data, which is generated from an import UDF using python libraries. The resultant data exists in memory only, not in a storage system. This connector type is not common; it is used mainly by advanced users for testing purposes.

Google Cloud Storage

 Google Cloud Storage Account with Environmental Credentials A Google Cloud Storage account that authorizes access from a Google services account authentication file whose location is set in the GOOGLE_APPLICATION_CREDENTIALS environment variable. For more information about Google Cloud account authentication for application Clients, see the Google documentation.

Apache Hadoop data lake

 HttpFs An Apache Hadoop Distributed File System (HDFS) data storage system that is accessed through the HttpFS protocol.

Apache Hadoop data lake

 HttpFs with Kerberos An HDFS data storage system that uses Kerberos authentication and is accessed through the HttpFS protocol.

Apache Parquet

 Parquet Dataset

When importing from a parquet dataset, Xcalar requires two connectors that work together to read the parquet dataset:

  • An existing connector that specifies the location of your parquet source data
  • The Parquet Dataset connector, which requires access to the first connector.

Amazon S3 storage

 S3 Account with Environmental Credentials

An Amazon S3 storage account, such as Amazon Web Services (AWS), that authorizes access from an AWS Boto3 configuration file that contains the following S3 storage authentication credentials:

  • aws_access_key_id
  • aws_secret_access_key
  • aws_session_token
  • region

Amazon S3 storage

 S3 Account with Specified Credentials

An Amazon S3 storage account that authorizes access from the S3 storage account authentication credentials. During the connector creation task you will be required to supply the values for these credentials.

MapR platform

 Secured MapR Cluster Account

A MapR cluster platform that authorizes access using authentication credentials from an authorized MapR user.

MapR platform

Secured MapR Cluster Account Impersonation

A MapR cluster platfrom that authorizes access from a MapR impersonation ticket.

Network File System

 Shared File System A network file system (NFS) mount that is shared between all nodes of the Xcalar cluster. For example, the Xcalar Root Directory.

Single node file system

 Single Node Filesystem A file system that is mounted on all nodes of the cluster, which enables you to access each node's source data individually, unlike a shared file system. When importing from a connector of this type, you must specify the node number followed by the path of the source data file. This enables Xcalar to import data only from the specified node.

Sharded data

 Symmetric Pre-sharded Filesystem

A file system that is mounted on all nodes of the cluster, which contain evenly distributed sharded data files after splitting a large file. Each file shares the same directory name on each node.

EXAMPLE: You split a source data file named airlines.csv into 4 files named airlines1.csv, airlines2.csv, airlines3.csv, and airlines4.csv, where the first two files are saved to the /airlinesData directory on node 0 and the second two files are saved to the /airlinesData directory on node 1. When a Symmetric Pre-sharded Filesystem connector is created for /airlinesData, all four files are available for import.

Apache Hadoop data lake

WebHDFS

An Apache Hadoop Distributed File System (HDFS) data storage system that is accessed through the WebHDFS protocol.

Apache Hadoop data lake

 WebHDFS with Kerberos An HDFS data storage system that uses Kerberos authentication and is accessed through the WebHDFS protocol.
