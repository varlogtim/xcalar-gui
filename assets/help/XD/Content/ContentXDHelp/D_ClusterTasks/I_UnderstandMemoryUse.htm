Understanding memory usage

The Xcalar architecture makes optimal use of system resources (such as memory, CPU, and network resources). Xcalar can optimize memory usage even on a heterogeneous cluster where server nodes have different amounts of memory installed.

About Xcalar Compute Engine's use of the memory hierarchy

Xcalar Compute Engine's distributed shared memory (DSM) architecture automatically uses and manages the tiers of a memory hierarchy, which are shown in the following list:

  • DRAM
  • Storage Class Memory (SCM)
  • Solid state drives (SSDs)
  • Hard disk drive (HDD)

Xcalar Compute Engine uses sophisticated policies to optimally distribute data and meta data across the cluster. It also uses demand paging to move data between main memory and lower tiers of the memory hierarchy. When the Xcalar cluster is low on memory, Xcalar writes data to lower tiers of the memory hierarchy. This process is called serialization. When more memory is available, Xcalar moves data from the demand paging area back to memory. This process is called deserialization. While serialization and deserialization take place without user intervention, Xcalar recommends that users conserve memory (with methods described in Optimizing memory usage) so that Xcalar Compute Engine can deliver the highest possible performance.

Xcalar is designed to automatically drop the result sets after certain period of inactivity. The result set can be regenerated by right clicking on a dataflow Node and selecting the Execute menu item. You can prevent the system from dropping a result set by right clicking on the dataflow Node and selecting Lock Result menu item.

Xcalar gives you fine-grained control over the distribution of data and meta data across the cluster. Such control is particularly useful when you have to deal with non-uniform data distribution or coloration that occurs for real-world applications. Production systems often need to run optimized dataflows in a similar way as database administrators optimize SQL queries. Contact Xcalar if you need Xcalar's Field Technology Engineers to help operationalize dataflows for you or provide training on operationalizing dataflows.

What happens when the cluster is low on memory

If the cluster is healthy without low-memory issues, is displayed next to the user icon in the top right corner of Xcalar Design. You can mouse over this icon to view the percentage of memory used. Click the icon to display the Statistics window, where you can see more details about resource usage. If Xcalar Compute Engine is low on memory, Xcalar Design displays a warning icon next to the user icon. A yellow icon indicates that memory consumption is at 60% and a red icon indicates that memory consumption is at 80%. A warning tooltip is also displayed, recommending the action to take for releasing memory.

Determining which result set to drop

Use the following list to determine which result set to drop:

  • From the Drop Result Sets modal window, you can see the amount of memory consumed by each result set. The size information helps you decide the amount of memory that will be released.
  • You might have multiple active result sets in your workbook. It is least likely that you want to drop active result sets because they are the ones that you are currently working with or worked with recently. However, it is acceptable to drop an active result set if it is not needed anymore. For example, because another active result set contains similar data.

Dropping a result set from the Drop Result Sets modal window

Follow these steps to drop result sets:

  1. To display the Monitor window, from the sidebar, click the Monitor icon ().
  2. Click Release Memory, which displays the Drop Result Sets modal window.

  3. In the Drop Result Sets modal window, select the result sets to drop.

    NOTE: Before selecting a result set, you can sort the result sets by name, size, or date modified.
  4. Click Drop and Confirm.
