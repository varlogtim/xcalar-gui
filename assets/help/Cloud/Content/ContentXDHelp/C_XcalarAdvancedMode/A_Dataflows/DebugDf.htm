Debugging a dataflow

This topic describes the best practices for developing dataflows, detecting issues, and debugging them. It assumes that you know how to create and edit a dataflow as described in Creating and editing a dataflow.

Best practices for developing a dataflow

Xcalar provides a lot of features for debugging your data models in the Xcalar Design UI. Here are some good practices to make it easier:

  • Build modular dataflows using the Link In and Link Out Nodes so that it is easier to isolate the problems.
  • Remove or hide the columns that you are not going to use. It reduces the clutter and improves the focus.
  • Missing data may produce unexpected results. Either remove the rows with missing values or fill them up with something meaningful.
  • Once you have developed your dataflow, add meaningful descriptions or comments to the Nodes for a quick recollection of functionality at a later time.

Detecting problems in a dataflow

Here are some general guidelines for detecting problems after the execution of a dataflow:

  • Check the status of every Node. If it is red, the Node generated an error.
  • If the icon of a Map Node appears red with an exclamation sign, the Node generated errors. Click on the red area of the Node to display a list of errors.
  • View the Node stats:
    • Compare the number of rows of each Node with the previous Node. If the number of rows is unusually different than your expectations, analyze the result sets.
    • If the time taken to execute a Node seems high, you might want to change the configuration of the Node or redesign the dataflow. For example, if a SQL Node is taking a long time to execute, you might want to rewrite the SQL snippet.
    • Look at the skew number of each Node. If the skew number is high, you might want to find out the reason. Joins with not enough number of unique key values is a common cause of high skew.

Debugging by analyzing a result set

Here are some general guidelines for analyzing your result set:

  • Sort the columns both in ascending and descending order to check for out of range values or FNFs.
  • Profile a column to check for unexpected values or to view the distribution of key values.
