Xcalar Glossary of Terms

The following table lists the Xcalar glossary of terms,

Table 2 Glossary of terms

 

 
Term Description
connector

The Xcalar metadata that describes the source data. The metadata includes your source data's format, storage location and repository type, and access authentication credentials and method.

Dataflow Mode

A workspace for developing and debugging data models using visual components, Xcalar SQL, and Python UDFs.
dataflow

A modeling algorithm created from a group of operations that retrieve, manipulate, and output data.

In Xcalar Design, a dataflow is displayed as a graphical diagram containing individual components that represent each phase of the algorithm. Where, the input, output, and operation phases are represented by configurable graphical components called Nodes.

Dataflow editor

A visual editor in Dataflow Mode for developing and debugging dataflows.

dataset

A representation in Xcalar Design of the raw source data, including the metadata created by Xcalar during the importing task. Depending on the value that is set by the Xcalar administrator for the maximum import size, the dataset may contain all or a sample of the raw source data.

Import your source data as a dataset:

  • When your data contains objects and arrays, as these data types are not supported in tables.
  • When you do not require a schema.
data source The file, directory, or database that contains your raw source data.
dropping a result set

The term for completely removing a result set or set of result sets, not just the contents, from a Xcalar workbook, to conserve memory. This action is not reversible.

NOTE: Dropping a result set is not the same as the DROP TABLE statement in SQL.
export The Xcalar task that saves a result set to a location outside of Xcalar Design.
import The Xcalar task that imports your data source into Xcalar into tables or datasets.
modeling

A set of data analysis tasks that retrieve, manipulate, and output data from a group of operations, which can result in the designing of an algorithm or a Xcalar dataflow.

node A bare metal computer or virtual machine that performs operations and communicates with other nodes of a cluster.

Node

A configurable graphical operation component of a dataflow in Xcalar Design.
operation A process that transforms data.
operationalization

The process of optimizing your dataflow for optimal performance and then deploying it on large scale data sets in a production environment.

In Xcalar Design, after building and debugging your dataflow, click Execute All Optimized, which:

  • Processes the dataflow to handle petabytes of data.
  • Adds parameterization, which inserts globally-defined parameters that enables you to run the dataflow on different types of source data.
query

A request for data or information from a table or tables. In Xcalar Design, you run Xcalar SQL SELECT query statements on your data, from either the SQL mode or the Dataflow Mode workspaces, to transform and retrieve information.

result set

The visual representation of the results from a SQL query statement, Node operation, or dataflow in the form of a table. The table includes metadata about the query, Node operation, or dataflow, such as column names, data types, table size, and creation time .

skew

The percentage value of the data distribution across the cluster nodes. Where a value of 0 denotes a balanced distribution of data across all nodes in the cluster and 100 denotes an imbalanced distribution of data, indicating that the data is residing on one node of the cluster.

Data skew can severely downgrade the performance of SQL queries. When data is distributed evenly across the Xcalar cluster nodes, the faster the operations will run. However, operations that use JOINS, and WHERE and HAVING clauses may require code changes that select only those columns with the most evenly distributed values.

SQL Mode

A workspace for developing data models using Xcalar SQL and Python UDFs.

SQL function

A dataflow created in Dataflow Mode that enables you to include table manipulation operations as a function in your SQL query statements. It contains a series of operations that manipulate and retrieve data from one or more tables (Input Nodes) and returns the results in another table (Output Node). The SQL function can replace any table within a SQL query statement.

NOTE: SQL functions can only have one output result. If you require multiple output results, you must create multiple SQL functions.
table

A representation in Xcalar Design of the raw source data, including the table's schema and the metadata created by Xcalar during the importing task.

Data in tables can be manipulated in both the SQL Mode and the Dataflow Mode workspaces.

NOTE: To develop your data models in the SQL Mode workspace, you must import your data into tables.
True Data in PlaceTM

Xcalar's True Data in PlaceTM technology keeps all source data in its original form on the native storage platform. Instead of moving or copying the data to memory, Xcalar creates metadata that references the source data. Relational operations are performed using this metadata and not on your source data. By avoiding the movement of data and performing analysis without creating copies, no vulnerabilities are introduced into your data stack or your data security procedures and protocols.
user-defined function (UDF)

A UDF is a function coded in Python by the Xcalar user for extending the functions of Xcalar. It generally accepts parameters, performs an operation or task on those parameters, and returns a result as either a singular value or as a result set. Xcalar provides a collection of built-in UDFs in the default.py module.

You can create additional UDFs to further extend the functions of Xcalar, such as:

  • Import UDFs, for parsing data in an import data format that is not supported by Xcalar.
  • Export UDFs, for processing data in an export data format that is not supported by Xcalar.
Workbook

The container for your Xcalar Design work.

Xcalar Compute Engine

A distributed runtime engine that processes data and performs relational operations on each Xcalar cluster node.

Xcalar Design

A browser based visual design studio for designing and creating data models that analyze data.

The Xcalar Design integrated development environment (IDE) has two workspaces:

  • SQL Mode
  • Dataflow Mode
Xcalar Processing Unit (XPU)

A process that runs custom logic on data, such as from a UDF, within its own domain and outside of the Xcalar Compute Engine.
