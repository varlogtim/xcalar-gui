Understanding the data source format requirements

To use Xcalar reliably and predictably, you must make sure that your data source meets the requirements described in this topic. You might need to prepare your data files so that Xcalar Design can properly import the data.

Data formats supported

Xcalar Design natively supports CSV, JSON, Excel, XML, raw text, and Parquet files. For other formats, you must create a user-defined function (UDF) for importing the data and instruct Xcalar Design to use that UDF during the import.

NOTE: The CSV format includes not only the comma-separated values but also the values separated by alternate delimiters. The default field delimiter is the tab (\t) character and the default record delimiter is the newline (\n) character. You can select the delimiters for separating the fields and for separating the records at the time of the import.

Verifying the file format or schema

The data source can be a file or a directory. If you import a directory, Xcalar Design imports the data from all the files in that directory, unless a search pattern is specified. You can also import the subdirectories recursively.

If you import a directory or multiple files, make sure that the source files meet the following requirements to avoid unpredictable results:

  • All files are in the same format. For example, they must be all JSON files or all XML files.
  • The files have the same schema.

Using the first row as column headers

NOTE: The information about using the first row as column headers is not applicable if the data format is JSON, XML, or user-defined.

Xcalar recommends that you use the first row of your data file as column headers. If it does not exist, add the first row to use as column headers. For information about naming conventions for column headers, see Preparing the first row as column headers. If you do not provide the column headers, Xcalar Design uses column0, column1, and so on, as the default column headers.

IMPORTANT: If you import a directory, the first row must be the same in each file in the directory. Xcalar Design processes data files in parallel so the order of the files being processed is not predetermined and you cannot rely on only the first file in the directory having the correct column headers row.

Column headers improve readability of the tables and the datasets created by Xcalar Design. In addition, if you use a user-defined function (UDF) for parsing the data while importing, be aware that after Xcalar Design imports the data, the columns may be ordered differently than the columns in the data source. If the columns do not have headers, you might not be able to determine the meaning of each column based on its position.

Example: Your data source is an Excel file with ID numbers in its first column. When you import an Excel file, Xcalar Design invokes a UDF to parse the data. If the file does not have headers, it might be difficult to determine which column is for ID numbers because the ID column may not be the first column after the data import. If, before you import data into Xcalar, you create a header row as the first row of the Excel file, and assign a name to the first column (for example, ID), then you can easily locate the column in Xcalar Design.

If you split a large source file into smaller ones for importing, remember to add the column headers row of the original file to all the smaller files. This ensures that the same first row in each source file can be promoted as column headers during the import.

Preparing the first row as column headers

Before you promote the first row of your data file as column headers, make sure that the first row of the source file does not contain any illegal characters. If there are illegal characters, edit the first row to remove them. Alternatively, you can import the data source by using the default user-defined function (UDF) provided by Xcalar to replace these characters.

The column headers must use the following naming conventions.

The following list describes the naming conventions:

To avoid any issues from the case sensitivity column naming differences between Xcalar and Xcalar SQL, column names are automatically processed into uppercase characters.

  • Length must not exceed 255 characters.
  • First character must be a letter of the alphabet or an underscore.
  • First character must not start or end with a space.
  • Characters must be A-Z, a-z, 0-9, or underscore (_).
  • Cannot contain special or punctuation characters, such as:
    • caret (^)
    • single (') or double (") quote marks
    • left (() or right ()) parenthesis
    • left ({) or right (}) brace
    • forward slash (/) or backslash (\)
    • colon (:) or semi-colon (;)
    • comma (,)
    • period (.)
    • single (-) or double hyphen (--)
  • Must not be one of the following reserved names:
    • Data
    • None
    • False
    • True

Using quoting characters in a CSV or a text file

If your data source is a CSV or a text file, check to see if they contain a double quote mark (") that should be interpreted literally. By default, Xcalar Design considers the double quote mark as a character to surround a string value. If the double quote mark in your data source should be parsed literally, you must override Xcalar Design's default by specifying no character as the Quoting Character during the import process.

Unicode encoding

Unicode is supported for both delimiter-separated values and JSON that is UTF-8 encoded. Import user-defined functions (UDFs) can convert other text encoding standards, like UTF-16, to UTF-8 without any loss of information.

Size limit for the import

There is a limit to how much data Xcalar Design can import from a single data source and that value is 10 GB.

Because Xcalar cannot create a table or a dataset by importing only a partial file:

  • If you want to import a single file, it must be under this limit.
  • If you want to import multiple files or a folder containing multiple files:
    • in a single dataset or a single table, the total of all the file sizes must be under this limit. Xcalar skips importing the files after this limit is reached.
    • as individual datasets, every file must be under this limit. Any file above this limit is skipped.
EXAMPLE: If you try to import a 12-GB file named airlines.csv. The attempt to import this file fails because Xcalar cannot create a table or a dataset by importing only a portion of the airlines.csv file.
EXAMPLE: If you try to import a folder containing a 2-GB file named airlines1.csv and a 12-GB file named airlines2.csv. Xcalar can successfully import the airlines1.csv file but no records from airlines2.csv.

To work around the problem of importing a large data source:

  • Split a large source data file into smaller files.
RECOMMENDATION: For better performance, split a large source data file into smaller files that are from 1 MB to 100 MB. Also, you can improve performance if all files are of similar sizes.
IMPORTANT:  The size limit takes effect only when you use Xcalar for modeling. When you execute a dataflow as optimized, the import process is not affected by this size limit.
