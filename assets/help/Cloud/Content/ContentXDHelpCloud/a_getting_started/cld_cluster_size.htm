Understanding Xcalar cluster sizes

This topic provides general guidelines for selecting a Xcalar cluster size.

Selecting a cluster size depends on various factors, such as:

  • The number and type of data sources
  • The size of your data
  • The complexity of your data model etc.

Due to the subjective nature of the factors involved, Xcalar cannot recommend a specific cluster size for your needs. However, Xcalar makes it easy for you to experiment with different cluster sizes and select the one that fits your needs.

Selecting a cluster size

Xcalar recommends that as a first time user, start with the smallest available cluster. Import your data files into the cluster and monitor its performance. If it appears slow, log out and select a bigger cluster size.

After the import, start building a data model. At any given point, if you find the performance to be slow, log out and select a bigger cluster size.

NOTE: There is no penalty for switching a cluster, however, every time you switch a cluster, you lose the intermediate results of your data model executions and you have to execute the data model from the beginning.

Billing for the Xcalar cluster

Some important notes regarding the billing of your Xcalar cluster:

  • You are billed only when the cluster is running.

    It is recommended that you stop the cluster when you are not using it. Xcalar Design also provides a setting to control the idle duration after which the user is automatically logged out and the idle cluster is automatically stopped. For more information on this setting, see Configuring general settings.

  • Xcalar uses per minute billing.

    This helps you in experimenting with different cluster sizes without wasting money on long billing cycles.

Impact of bigger cluster size

In general, increasing the cluster size improves the performance of your data model because bigger cluster size gives you more compute power. However, smaller data models may not necessarily perform better on a bigger cluster.

Selecting a bigger cluster means more expense per minute but it may not necessarily be more expensive overall because you may be able to develop and execute your data model faster, thus spending less money.
