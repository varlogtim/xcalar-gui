Optimizing memory usage

Xcalar Insight displays a low-memory notification to all logged-in users when XCE reaches a pre-defined level of memory consumption. The Xcalar administrator might also notify users to free up memory before the automatic notification is sent. It is important for you to optimize memory usage when using Xcalar Insight to prevent the cluster from running out of memory. See Understanding memory usage for more information about the low-memory notification and what steps to take in response to the notification.

Best practices for conserving memory

Xcalar Insight users can follow these best practices to conserve memory:

  • Reduce the number of tables.

  • Reduce the number of fields in the Data Browser.

    NOTE: Fields that come from your dataset cannot be removed individually. For example, if your dataset has two fields: Carrier and Description, you cannot remove only the Carrier field and keep the Description field to try to reduce the number of fields.

More detailed information about these best practices are provided in the sections that follow.

Effects of tables on memory consumption

Each table in your workbooks consumes memory regardless of the table's status, which can be active, archived, or orphaned. Keeping only the tables required for modeling your queries helps conserve memory.

Methods for deleting tables

You can delete tables with these methods:

  • Display the Tables panel. Then select the tables to be deleted. See Understanding and changing table statuses for more information about the Tables panel.
  • In the query graph, right-click a table icon to display a pop-up menu. Select Delete Table to remove the table from the worksheet. This method enables you to see how the table was created or modified when you performed the modeling.
  • In the active worksheet, locate the table to delete, and right-click the table title bar to display a drop-down menu. Then select Delete Table.
  • In the Monitor tab, click the Delete Tables button to display the Delete Tables modal window, which lists all tables in your workbook and the amount of memory used by each . See Understanding memory usage for more information about the Delete Tables modal window.

Effects of table columns on memory used

The number of columns in a table has no significant effect on memory consumption. For example, from the same dataset , you can create two tables: one with 2 columns and one with 15 columns. Both tables use the same amount of memory, which is 2.2 MB.

Because the number of columns is not correlated with memory consumption, you cannot free up memory by deleting table columns.

Effects of fields in the Data Browser on memory used

Each table has a Data Browser associated with it. The Data Browser lists all the fields that are already in the table or can potentially be added to the table. Initially, each table consumes the same amount of memory as each table's Data Browser contains only the fields from the raw dataset. This collection of fields from the dataset consumes a fixed amount of memory, regardless of how many fields are in the group.

After you perform operations on a table, derived columns are created in the resultant table and these columns are also added in the resultant table's Data Browser as new fields. Each of these new fields consumes additional memory.

Example: A two-column table is created from a dataset with two fields: Destination and Distance. If no operations have created columns in this table, the table uses 2.2 MB of memory. Suppose you change the Distance column's data type to integer. A new table is created with a derived column named Distance_integer. In the Data Browser for this new table, the field Distance remains, and a new field named Distance_integer is added at the end of the field list. If you use the Delete Tables modal window to check memory usage, you can see that the new table consumes 4.2 MB instead of 2.2 MB of memory because of the derived field added to the Data Browser. The more derived columns your operations create, the more derived fields are added to the Data Browser, which increases the amount of memory consumed.
NOTE: As in the case of other columns, deleting a derived column does not free up memory. For example, if you delete the Distance_integer column in the example above, the table continues to use 4.2 MB of memory. Memory used is not reduced because the derived field, Distance_integer, remains in the Data Browser.

Removing fields from the Data Browser

Because the number of fields in the Data Browser has an impact on memory used, the logical step to conserve memory is to reduce the number of fields in the Data Browser. However, remember that in the context of memory consumption, the collection of fields from the source dataset is considered one unit. You cannot remove some fields and keep some fields in that collection. You can decide either to keep or to eliminate the collection in its entirety.

The contents of the Data Browser are not directly editable. To change the contents, use the Projection Mode feature, which enables you to keep only columns of interest in a table and to remove unnecessary fields from the Data Browser.

NOTE: The term "projection" originates from relational algebra. Its meaning in Xcalar Insight should be clearly distinguished from its meaning in daily usage.

Understanding Projection Mode

Projection Mode in the Data Browser enables you to eliminate unnecessary fields that Xcalar Insight keeps in memory for a table. You can choose the fields that interest you to keep both in the table and the Data Browser. In doing so, you can reduce the amount of memory used by the table.

Using Projection Mode

Follow these steps to use Projection Mode to keep memory consumption low for a table:

  1. Double click anywhere in the DATA column of the table to display the Data Browser.
  2. Click the down arrow in the Data Browser to select Projection Mode.

  3. In the list of fields, click the field that you want to keep in the resultant table. For example, click Distance_integer if you want to keep it in the resultant table. You might have to use the scrollbar to view all field names. Derived columns are located at the end of the list.

  4. Click the submit projection button. A new table is created with only the columns you chose in the Data Browser. As in the case of other operations, you can use the Undo button to revert the table to the previous version before the projection.

You can perform projections on any columns to reduce the number of columns in the active table so that the table is easier to use. However, for the purpose of optimizing memory usage, pay attention to the number of fields being eliminated from the Data Browser when you decide which column to choose for projection.

Example of projection on derived fields

The Data Browser shows 34 fields. The last 19 fields on the list are derived fields such as Distance_integer, DepTime_integer, DepTime_integer_gt62, and so on. These derived fields are created due to operations such as Map. The rest (15 fields) are from the collection of fields in the source dataset. They are MonthDayYear, Distance, DepTime, and so on.

Suppose you perform a projection on Distance_integer and DepTime_integer, the resultant table contains only these columns. In addition, all fields except for Distance_integer and DepTime_integer are eliminated from the Data Browser. The resultant table now consumes much less memory than the table before the projection. For the resultant table, the XCE cluster no longer needs memory to track 17 derived fields and the collection of fields from the source dataset.

Example of projection on fields from the source dataset

You can choose any field for projection. If one or more fields chosen are from the collection of fields that exist in the source dataset, you will notice that the entire collection remains in the Data Browser. This is because this group of fields can be removed or kept as a unit. It consumes a fixed amount of memory, which is about the same as that consumed by a derived field.

Suppose you perform a projection on Distance and MonthDayYear, the resultant table contains only these two columns. The Data Browser keeps the entire collection of fields from the source dataset. While the Data Browser shows 15 fields in this collection, the projection actually frees up memory that otherwise would be needed to track the 19 derived fields.

To conclude, fields that are projected are kept in the resultant table and its Data Browser; all others are eliminated. The memory savings due to projection depend on the number of fields the projection eliminates in the Dataset Browser. When considering the number of fields being eliminated, remember that the group of fields from the dataset is kept or removed as a unit, and this unit consumes the same amount of memory as a derived field.

The Xcalar administrator might also notify users to free up memory before the automatic notification is sent. It is important for you to optimize memory usage when using Xcalar Insight to prevent the cluster from running out of memory resources.
